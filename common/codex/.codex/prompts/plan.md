# plan

> 本文件是本地工程编排里的“计划节点”。
>
> 它的职责不是把所有任务都变成长计划，而是先决定当前任务需要哪种计划深度：
>
> - 轻量执行说明
> - 分阶段实现计划
> - 升级为 `AGENT_SPEC.md` 的 task contract

## 先看哪里

在使用本提示前，先参考：

- `~/.codex/ENGINEERING.md`
- 若任务风险高或验收复杂，再参考 `~/.codex/AGENT_SPEC.md`

## 什么时候使用

以下情况优先进入计划节点：

- 新功能
- 中高风险重构
- 需求含混，需要先收敛边界
- 多文件/多模块改动
- 需要判断是否应该立 spec

以下情况通常不需要重计划：

- 纯问答
- 极小改动且影响面清晰
- 已有明确 contract/spec，只需执行

## 计划深度决策

先判断任务属于哪一层：

### 轻量执行说明

适用于 `L0-L1`：

- 单文件或局部改动
- 外部行为变化很小
- 风险和回滚路径都清楚

输出：

- 目标
- 约束
- 预计改动点
- 验证方式

### 分阶段实现计划

适用于 `L2`：

- 多文件或多模块
- 需要拆阶段、理依赖、控制风险

输出：

- 需求重述
- 决策与约束
- 阶段拆分
- 风险与回滚考虑
- 验收方式
- 推荐后续工作流

### Task Contract

适用于 `L3`，或满足以下任一条件：

- API / schema / migration / security / auth / payment
- 验收标准复杂
- 需要明确 allowed / forbidden changes
- 需要机械验证和高可审计性

这时不要只写普通计划，转而按 `AGENT_SPEC.md` 产出 task contract。

## 计划输出模板

无论深浅，至少覆盖以下内容：

### 1. Outcome

- 要达成什么结果
- 用户真正关心的行为变化是什么

### 2. Constraints

- 不该改什么
- 不能破坏什么
- 有哪些时间、依赖、兼容、性能或安全约束

### 3. Decisions

- 已经固定的技术/实现决策
- 哪些地方不允许边做边改主意

### 4. Boundaries

- 预计涉及哪些文件、模块、接口
- 哪些部分明确 out of scope

### 5. Acceptance

- 什么证据说明完成
- 需要哪些测试、验证或 review

### 6. Risks

- 正确性风险
- 回归风险
- 运行/部署/数据风险

### 7. Route

计划末尾必须明确下一步：

- 直接实现
- 先写 task contract
- 先补充澄清
- 先做调查/复现

## 输出风格

优先使用简洁、可执行的内容，不堆模板话术。

计划应该帮助后续执行，而不是制造新的文档负担。

好的计划应该让人一眼看出：

- 为什么这样做
- 为什么不那样做
- 先做什么
- 什么时候停下来验证

## 与其他提示的关系

- 计划完成后，实现默认进入 `tdd.md`
- 如果任务本质是 bug，先用 `debug.md` 再决定是否需要计划
- 计划中的局部检查走 `quality-gate.md`
- 完成前总验证走 `verify.md`
- 高风险改动在完成后补 `code-review.md`

## 明确等待

如果用户明确要求“先给计划，不要动代码”，则：

- 输出计划后停止
- 等用户确认、修改或缩范围

如果用户没有要求停在计划阶段，则：

- 把计划当作内部路由决策
- 继续进入实现或调查
