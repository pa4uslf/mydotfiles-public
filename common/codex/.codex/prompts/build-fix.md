# build-fix

> 本文件是本地工程编排里的“构建修复节点”。
>
> 它适合处理 build/type/lint 层面的可机械修复问题，但不应该借修构建之名做未计划的大改动。

## 什么时候使用

优先用于：

- build 失败
- typecheck 失败
- 一组清晰、可枚举的编译错误
- 局部 lint/static analysis 问题需要成批清理

以下情况先不要直接用本提示硬修：

- 根因明显是业务 bug 或回归 -> 先 `debug.md`
- 修构建会牵出架构重做 -> 先 `plan.md`
- 缺依赖、缺环境、缺外部服务 -> 先确认环境前置条件

## 目标

用最小、安全、可回归验证的方式，把仓库从“不能构建/不能通过静态检查”恢复到健康状态。

## 默认流程

### 1. 识别真实失败入口

优先识别项目实际使用的：

- build
- typecheck
- lint

不要凭空猜命令名。

### 2. 先跑一次完整失败

记录：

- 哪个命令失败
- 错误总量
- 首个失败点
- 是否存在“一个根因引发大量派生错误”

### 3. 分组错误

按以下顺序优先处理：

1. 语法 / import / symbol 缺失
2. 类型定义 / contract 不一致
3. 派生的实现错误
4. 次要 lint / style 问题

避免逐条盲修明显由同一根因导致的报错洪流。

### 4. 一轮只修一类根因

每轮修复都要：

1. 读上下文
2. 判断根因
3. 做最小修复
4. 重跑相关命令
5. 确认错误数量确实下降

### 5. 局部闸门

每轮修复后，至少跑：

- 相关 build/type/lint 命令
- 必要时 `quality-gate.md`

如果仓库已恢复健康且接近交付，再进入 `verify.md`。

## 护栏

出现以下情况要停止“继续硬修”的冲动，并升级流程：

- 同一错误 3 轮仍未消失
- 修一个地方引出更多结构性问题
- 需要大规模重构才能通过构建
- 问题根因其实是错误设计、错误依赖边界或错误需求

对应升级：

- 根因不明 -> `debug.md`
- 结构性调整 -> `plan.md`
- 完成前总检查 -> `verify.md`

## 结果输出

至少说明：

- 跑了哪些命令
- 修掉了哪类错误
- 还剩哪些问题
- 是否已恢复到可继续开发/可交付状态

## 原则

- 最小修复优先于顺手重构
- 先修根因，再修噪音
- 不要为了“全绿”偷偷改变外部行为
- 如果行为已经被影响，补测试或回到 TDD / debug 流程
