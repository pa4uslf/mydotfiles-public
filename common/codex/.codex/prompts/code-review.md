# code-review

> 本文件是本地工程编排里的“评审节点”。
>
> 默认采用 findings-first 的 code review 风格：先指出问题，再给摘要。

## 目标

对当前 diff 或指定范围做一轮面向风险的评审，优先发现：

- 正确性问题
- 行为回归
- 安全问题
- 缺失测试或验证空洞
- 与计划 / contract / 边界不一致之处

## 评审对象

默认评审当前未提交变更；如果用户给定范围，则以用户范围为准。

先获取：

- changed files
- 关键 diff
- 涉及的测试与验证结果
- 如存在，计划或 spec

## 评审优先级

### CRITICAL

- 明显安全漏洞
- 数据损坏/丢失风险
- 权限绕过
- 支付/鉴权/敏感数据处理错误
- 会导致生产事故的明显缺陷

### HIGH

- 逻辑错误
- 回归风险高
- 关键异常路径未处理
- 契约/API/schema 被破坏
- 改动没有必要的测试支撑

### MEDIUM

- 可维护性问题
- 边界不清
- 抽象错误或扩散过度
- 性能/可访问性等中等级风险

### LOW

- 可读性、命名、局部一致性

## 评审清单

### 正确性

- 代码是否真的实现了目标行为
- 边界条件是否成立
- 错误路径是否被处理
- 状态转换是否完整

### 回归风险

- 是否破坏既有调用方
- 是否更改了未声明的行为
- 是否引入时序、缓存、一致性问题

### 安全

- secrets / credentials / tokens
- 输入验证
- 注入、XSS、路径遍历
- 鉴权、授权、敏感数据处理

### 验证完整性

- 新行为是否有证据
- bug 修复是否有回归测试
- UI/用户流是否需要 E2E
- `verify` 的结论是否覆盖了关键路径

### 计划一致性

如果任务有计划或 spec，要检查：

- 是否偏离已定决策
- 是否越界修改 forbidden areas
- 是否漏掉了验收要求

## 输出格式

输出必须：

- findings-first
- 按严重级排序
- 给出文件和行号
- 简明说明影响与建议

格式：

```text
Findings
1. [CRITICAL/HIGH/MEDIUM/LOW] path:line - issue and impact
2. ...

Open Questions / Assumptions
- ...

Summary
- optional short summary only after findings
```

## 阻断规则

若发现 `CRITICAL` 或 `HIGH`：

- 默认结论是不建议合入/提交
- 必须明确写出阻断原因

若未发现问题：

- 明确写“未发现阻断问题”
- 但仍指出验证缺口或残余风险
