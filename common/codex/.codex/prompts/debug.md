# debug

> 本文件是本地工程编排里的“调试入口”。
>
> 默认采用 root-cause-first，而不是 patch-first。

## 目标

把“修 bug”拆成一套稳定流程：

1. 复现
2. 缩小范围
3. 建立根因假设
4. 用失败证据锁定问题
5. 最小修复
6. 验证回归与副作用

## 使用场景

优先用于：

- 用户报告 bug
- 测试失败
- 线上/本地回归
- “昨天还好好的”
- 报错、异常、状态不一致

## 默认流程

### 1. Reproduce

先让问题稳定复现。

可接受的复现证据：

- failing test
- 最小脚本
- 可重复命令
- 浏览器复现步骤
- 清晰日志或错误栈

如果无法复现，不要直接修。

### 2. Narrow

缩小问题范围：

- 哪个模块开始出错
- 哪次状态转换开始偏离
- 哪个输入触发异常
- 哪些改动可能相关

### 3. Hypothesize

提出根因假设，并让假设可证伪。

一个好的假设应该说明：

- 为什么会错
- 为什么只在这类场景错
- 为什么以前可能没暴露

### 4. Lock With Evidence

在改代码前，尽量把问题锁定成失败证据：

- 回归测试
- 断言
- 复现脚本
- 差异化日志

目标是让修复前后差异可见，而不是凭感觉判断。

### 5. Fix the Smallest Thing

只修改导致问题的最小必要代码。

禁止：

- 没证据时大面积试错
- 顺手大改 unrelated 结构
- 用更多分支掩盖根因

### 6. Verify

修复后必须确认：

- 原问题消失
- 回归测试通过
- 相邻关键路径没被带坏
- 如有必要，进入 `verify.md`

## 输出要求

调试过程中要能说明：

- 复现方式
- 根因结论
- 修复点
- 如何验证
- 剩余风险

## 与其他节点的关系

- 需要结构化调查时，可结合 `gstack-investigate`
- 修复实现时，进入 `tdd.md`
- 局部检查走 `quality-gate.md`
- 完成前总验证走 `verify.md`
