# add-tests

> 本文件是本地工程编排里的“定向补测节点”。
>
> 它的职责不是机械地给 touched files 补单测，而是根据这次改动真正缺失的证据层，补最小但足够的测试。

## 什么时候使用

优先用于：

- 新代码已经写完，但测试覆盖明显不足
- bug 已修复，需要补回归测试
- review / verify 明确指出“缺测试”
- 想补关键行为证据，但不需要完整重走一次 TDD

如果实现尚未开始，优先走 `tdd.md`，不要把 `add-tests` 当成 TDD 的替代品。

## 目标

补的不是“测试数量”，而是“风险证据缺口”。

先判断缺的是哪一层：

- unit
- integration
- E2E
- 最小复现实验 / 回归脚本

## 默认流程

### 1. 明确行为变化

先识别：

- touched files 改变了什么行为
- 哪些是 public API / contract / 用户路径
- 哪些边界条件最容易回归

### 2. 选择最小需要的测试层

默认规则：

- 纯函数 / 局部逻辑 -> 先补 unit
- 模块协作 / API / 数据流 -> 至少补 integration
- 浏览器交互 / 多步骤用户流 -> 评估 E2E
- bug 修复 -> 必须先有失败复现或回归测试

### 3. 先写失败证据

要求：

- 测试名说明具体行为
- 覆盖 happy path
- 覆盖至少一个关键 edge/error path
- bug 修复必须覆盖原始失败场景

### 4. 保持 diff 最小

- 使用项目已有测试框架和目录约定
- 不引入不必要的新测试工具
- 只补本次改动真正需要的测试

### 5. 跑目标测试

至少运行：

- 新增/修改的测试
- 直接受影响的现有测试

必要时升级到：

- `quality-gate.md`
- `verify.md`

## 选择标准

### 优先补 unit

当问题局限在：

- 纯计算
- 数据转换
- 条件分支
- 小范围状态逻辑

### 优先补 integration

当问题涉及：

- service + repository
- API handler + validation
- 多模块组合行为
- 配置/依赖注入后的真实行为

### 优先补 E2E

当风险在：

- 用户关键路径
- 浏览器状态切换
- 多步提交流程
- 前后端联动行为

## 输出要求

说明：

- 补了哪一层测试
- 为什么选这一层
- 覆盖了哪些行为
- 运行了哪些命令

## 与其他节点的关系

- 还没实现前，优先 `tdd.md`
- 根因不明的 bug，先 `debug.md`
- 局部检查走 `quality-gate.md`
- 接近交付时走 `verify.md`
